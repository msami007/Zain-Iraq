the# Zain Iraq Knowledge Base

## Pre-Demo UX Design Critique

**Three-persona walkthrough:** Frontline Agent → Admin / Content Creator → Super Admin

**Target:** https://zain-kb-app.onrender.com

**Prepared:** June 25, 2026

**Methodology note:** This critique was produced by logging into the live application as each persona via the Quick Demo Login buttons and performing the actual workflows a user in that role would perform, not by inspecting a static design file. Every finding below was reproduced and, where the result was surprising (e.g. a navigation failure or a data-isolation issue), cross-checked with a second method — a different click technique or a direct DOM read — before being recorded. One item (mobile responsiveness, see Cross-Cutting Observations) could not be fully re-confirmed before the testing session ended and is flagged accordingly rather than overstated.

# PERSONA 1: FRONTLINE AGENT

Walkthrough: logging in as a Zain/OODI Agent, searching the Knowledge Base under simulated call pressure, using the KB Assistant during a live chat, reporting a knowledge gap, and navigating between search results and article detail.

### F1.1 — Back-link from article detail discards search context

* **PERSONA:** Frontline Agent
* **MOMENT:** Mid-call, the agent has found a relevant article and wants to return to the result list to compare alternatives.
* **SCREEN-ELEMENT:** ‘← Agent Desk’ back-link at the top of the article detail view.
* **WHAT HAPPENS:** Clicking it navigates to the Dashboard, not back to the Knowledge Base result list the agent came from. The search query and result set are lost; the agent must reopen Knowledge Base and retype the search.
* **WHAT SHOULD HAPPEN:** The link should read ‘← Back to results’ when entered from search, and use history-aware routing (or pass the query as a URL parameter) so it restores the exact result set and scroll position.
* **IMPACT:** Every ‘let me check one more article’ click during a live call costs the agent a full re-search — a repeated time tax on every multi-article lookup, and a break in the basic mental model that ‘back’ means back.
* **SEVERITY:** HIGH

### F1.2 — Search match-score badge is meaningless

* **PERSONA:** Frontline Agent
* **MOMENT:** Agent searches the KB for a customer's issue and needs to judge at a glance which result actually matches.
* **SCREEN-ELEMENT:** Percentage ‘match’ badge on each search result card (e.g. ‘10% match’).
* **WHAT HAPPENS:** Every result for every query tested — over 50 distinct queries in English and Arabic, including nonsense strings like ‘qwertyzzz nonexistent xyz 12345’ — returns a near-identical low score (≈10–24%). The score never discriminates a perfect match from an irrelevant one.
* **WHAT SHOULD HAPPEN:** Fix the underlying relevance scoring so percentages vary meaningfully and irrelevant queries score near 0%, or remove the badge entirely until scoring is reliable.
* **IMPACT:** Agents are trained to use this number to triage results under time pressure. A badge that always reads the same regardless of fit teaches agents to ignore it — defeating its purpose on every single search.
* **SEVERITY:** CRITICAL

### F1.3 — Arabic-language search cannot find Arabic content

* **PERSONA:** Frontline Agent
* **MOMENT:** Agent receives an Arabic-speaking customer and searches the KB using Arabic text.
* **SCREEN-ELEMENT:** Knowledge Base search box and result list.
* **WHAT HAPPENS:** Typing the exact Arabic title of an existing article (دليل تفعيل شريحة زين) returns 47 results, all in English, all tagged ≈10% match, none of which is the article itself. Browsing the most likely categories by hand also failed to surface it.
* **WHAT SHOULD HAPPEN:** Match Arabic queries against Arabic content directly (or cross-match via transliteration), and rank an Arabic result above generic English filler when the query is in Arabic.
* **IMPACT:** Zain Iraq's customer base is majority Arabic-speaking; an agent fielding an Arabic-speaking customer currently cannot find Arabic support content through search at all.
* **SEVERITY:** HIGH

### F1.4 — Knowledge Gaps KPI tile uses the wrong trend direction

* **PERSONA:** Frontline Agent / Admin / Super Admin (same widget on all three dashboards)
* **MOMENT:** Glancing at the dashboard at the start of a shift to gauge backlog health.
* **SCREEN-ELEMENT:** ‘Knowledge Gaps’ metric tile (‘45 new this week ▼’), confirmed identically on the Zain Admin, OODI Admin, and Super Admin dashboards.
* **WHAT HAPPENS:** The tile shows a downward red arrow next to ‘45 new this week,’ which visually reads as ‘gaps are decreasing’ — but 45 new gaps is an increase in unresolved work. Elsewhere on the same dashboard, a down arrow on ‘Total Searches’ correctly signals something bad, so the same styling means opposite things on two tiles.
* **WHAT SHOULD HAPPEN:** Map arrow direction and color to the metric's real-world meaning, not its raw sign: a rising gap count should use an attention-getting indicator distinct from a tile where ‘down’ is genuinely good news.
* **IMPACT:** At-a-glance trend reading — the entire purpose of a KPI tile — cannot be trusted, and the bug is systemic across all three dashboards rather than a one-off.
* **SEVERITY:** MEDIUM

**What's working:** the ‘Copy to Chat’ action inside the KB Assistant panel inserts the article's answer and confirms with an immediate success toast — the kind of low-friction, unambiguous confirmation the rest of the product should match. The (FALLBACK) tab labeling on channel-variant content also persists correctly and communicates its purpose clearly.

# PERSONA 2: ADMIN / CONTENT CREATOR

Walkthrough: logging in as Zain and OODI Admins, drafting and submitting an article for review, hitting governance guardrails (team assignment, separation-of-duties self-approval blocking), and checking org isolation across Articles Manager, Workflows, Gaps Queue, Audit Logs, and Analytics.

### F2.1 — Saving with a missing required field fails silently

* **PERSONA:** Admin / Content Creator
* **MOMENT:** Admin clicks ‘Save Changes’ on a new article with the Title field left blank.
* **SCREEN-ELEMENT:** ‘+ New Article’ form, ‘Save Changes’ button.
* **WHAT HAPPENS:** Nothing happens: no error message, no field highlight, no toast. The admin cannot tell whether the click failed, is still processing, or saved a blank-titled article.
* **WHAT SHOULD HAPPEN:** Inline validation should highlight the Title field in red with ‘Title is required’ the instant Save is attempted, matching the pattern the app already uses for the team-assignment error.
* **IMPACT:** A new admin under demo pressure will click Save repeatedly assuming a bug, or wrongly believe a blank-titled article was saved.
* **SEVERITY:** HIGH

### F2.2 — Approve fails silently when self-approval is blocked

* **PERSONA:** Admin / Content Creator
* **MOMENT:** An admin who authored an article opens Articles Manager and clicks ‘Approve’ directly from the list view on their own pending article.
* **SCREEN-ELEMENT:** ‘Approve’ action button in the Articles Manager table row.
* **WHAT HAPPENS:** Clicking ‘Approve’ does nothing visible — no toast, no modal, no status change; the article silently stays ‘Pending.’ The real reason (separation-of-duties: authors cannot approve their own work) is only revealed one click deeper, on the full review screen.
* **WHAT SHOULD HAPPEN:** Show the separation-of-duties message at the point of the failed click (toast or inline tooltip on the list-view button), not only on a screen the admin has to navigate to separately.
* **IMPACT:** This is a silent failure on a core governance control. An admin could reasonably conclude the button is broken, file a bug, or assume the article actually published.
* **SEVERITY:** CRITICAL

### F2.3 — Stale error banner persists after the underlying issue is fixed

* **PERSONA:** Admin / Content Creator
* **MOMENT:** Admin submits without a team assigned, sees ‘At least one team must be assigned to the article,’ fixes it via the Assign to Teams dropdown, and saves again successfully.
* **SCREEN-ELEMENT:** Red error banner on the article edit screen.
* **WHAT HAPPENS:** After the article is confirmed re-saved and moved to Pending, the original red error banner remains on screen describing a problem that no longer exists.
* **WHAT SHOULD HAPPEN:** Clear error banners automatically once their condition is resolved and the form is successfully resubmitted.
* **IMPACT:** A stale error banner next to a successfully saved article makes the admin doubt whether the save worked, and would read as a live bug in front of a client.
* **SEVERITY:** MEDIUM

### F2.4 — ‘Pending’ and ‘In Review’ are the same status with two names

* **PERSONA:** Admin / Content Creator
* **MOMENT:** Admin cross-checks the Articles Manager status filter tabs against the Workflows pipeline labels for the same article.
* **SCREEN-ELEMENT:** Articles Manager filter tabs (‘Pending’) vs. Workflows ‘Editorial Pipeline’ stage labels (‘In Review’) vs. the underlying status value, which is literally ‘InReview.’
* **WHAT HAPPENS:** The identical content status is surfaced under two different labels depending on which screen the admin is looking at.
* **WHAT SHOULD HAPPEN:** Pick one term and use it everywhere: filter tabs, status dropdown, pipeline diagram, and audit log entries.
* **IMPACT:** A content team coordinating publishing day-to-day will think these are two different states, or look in the wrong filter tab for an article sitting right there under the other label.
* **SEVERITY:** MEDIUM

### F2.5 — Analytics Dashboard leaks cross-org search data into OODI's tenant-scoped view

* **PERSONA:** Admin / Content Creator (org isolation)
* **MOMENT:** Logged in as the OODI Admin, opening the Analytics Dashboard to review OODI's own search activity.
* **SCREEN-ELEMENT:** ‘Top Search Queries’ table on the OODI Admin Analytics Dashboard.
* **WHAT HAPPENS:** The table surfaces queries referencing ‘Zain app,’ ‘Zain SIM,’ and a test string (‘heheboi’) that do not belong to OODI's own 1-article tenant. By contrast, OODI's Articles Manager, Workflows, Gaps Queue, and Audit Logs are all correctly scoped — OODI cannot see Zain's pending article anywhere else in the product.
* **WHAT SHOULD HAPPEN:** Filter ‘Top Search Queries’ by tenant exactly as every other OODI screen does. If it is intentionally platform-wide, label it as such rather than nesting it inside OODI's own analytics view.
* **IMPACT:** This is the most damaging finding for enterprise credibility: it looks exactly like a cross-tenant data leak, in the one place — an org's own search analytics — where isolation matters most.
* **SEVERITY:** CRITICAL

### F2.6 — Audit log misattributes a customer action to the viewing admin

* **PERSONA:** Admin / Content Creator
* **MOMENT:** Admin reviews the System Audit Logs to confirm a customer-submitted knowledge gap was logged correctly.
* **SCREEN-ELEMENT:** Audit Logs table, ‘Actor’ column for a ‘KNOWLEDGE GAP REPORTED BY CUSTOMER’ entry.
* **WHAT HAPPENS:** Despite the action label saying ‘reported by customer,’ the Actor column shows ‘Admin (OODI) (admin.oodi@oodi.com)’ — the logged-in admin, not the customer who triggered it.
* **WHAT SHOULD HAPPEN:** Attribute the Actor column to whoever actually performed the action (Customer/Guest, as correctly shown in the Gaps Queue's own ‘Reported By’ column), not the account currently viewing the page.
* **IMPACT:** An audit trail that misattributes customer actions to staff accounts is a compliance and trust problem for an enterprise client evaluating governance.
* **SEVERITY:** MEDIUM

**What's working:** org isolation is correctly enforced on Articles Manager, Workflows, Gaps Queue, and Audit Logs — OODI genuinely cannot see Zain's content or activity on any of those four screens. The separation-of-duties rule itself is also a sound governance feature; the problem documented above (F2.2) is purely that its failure is silent, not that the rule is wrong.

# PERSONA 3: SUPER ADMIN

Walkthrough: logging in as Salman (Super Admin), attempting to reach platform-level administration (Organizations, User Accounts, Teams Manager) from the dashboard, and reviewing the platform-wide Analytics Dashboard.

### F3.1 — Super Admin sidebar navigation is completely non-functional

* **PERSONA:** Super Admin
* **MOMENT:** Salman logs in and tries to open ‘Organizations’ to manage the Zain/OODI tenants — the core reason the Super Admin role exists.
* **SCREEN-ELEMENT:** Entire left sidebar (Organizations, User Accounts, Teams Manager, Articles Manager, Gaps Queue, Workflows, Audit Logs, Glossary).
* **WHAT HAPPENS:** Clicking any sidebar item — verified individually on all eight — does not navigate anywhere. The Super Admin is permanently stuck on the Analytics Dashboard. Confirmed three independent ways: mouse-coordinate clicks, accessibility-tree reference clicks, and a DOM read confirming the buttons are real, present, clickable elements.
* **WHAT SHOULD HAPPEN:** Every sidebar item must route to its page, exactly as the equivalent sidebar already does correctly for the Zain Admin and OODI Admin roles.
* **IMPACT:** A complete, role-breaking functional failure, not a polish issue. The highest-privilege persona in the product can only ever look at one screen.
* **SEVERITY:** CRITICAL

### F3.2 — Platform-wide confidence metric reflects the same broken scoring, with no diagnostic context

* **PERSONA:** Super Admin
* **MOMENT:** Salman checks the platform-wide Analytics Dashboard to assess search quality across both tenants before a client call.
* **SCREEN-ELEMENT:** ‘Avg. Confidence’ tile (23.9%) and the ‘Recent Searches’ confidence column (10%, 21.7%, 17%).
* **WHAT HAPPENS:** The same broken match-scoring seen at the Agent level (see Persona 1) rolls up into a platform-wide number that reads as a clear red flag, with no explanation or drill-down for why confidence is low or what to do about it.
* **WHAT SHOULD HAPPEN:** Fix the underlying scoring (see Persona 1 finding); separately, add a drill-down or tooltip explaining what ‘confidence’ measures and what a healthy range looks like.
* **IMPACT:** Without diagnostic context, ‘23.9% confidence’ is just an alarming, unexplained number on the one dashboard meant to project executive confidence.
* **SEVERITY:** HIGH

### F3.3 — Viewing the Analytics Dashboard inflates its own ‘Total Views’ metric

* **PERSONA:** Super Admin
* **MOMENT:** Salman reloads the Analytics Dashboard to re-check a number.
* **SCREEN-ELEMENT:** ‘Total Views’ metric tile.
* **WHAT HAPPENS:** Total Views increased from 61 to 62 purely from reloading the internal Analytics Dashboard — viewing the metrics page itself counts as a customer-facing ‘article view.’
* **WHAT SHOULD HAPPEN:** Internal dashboard page loads must never increment customer-facing engagement metrics; view-counting should be scoped strictly to actual public article reads.
* **IMPACT:** Small in isolation, but it casts doubt on every other number on the dashboard — right as Salman would be using them to vouch for the platform to a client.
* **SEVERITY:** MEDIUM

# CROSS-CUTTING DESIGN OBSERVATIONS

Observations spanning all three personas: visual consistency, empty/error/success state coverage, mobile responsiveness at a 375px viewport, and an RTL/Arabic check.

### C1 — Empty states use three different visual patterns for the same situation

* **PERSONA:** Cross-cutting (all roles)
* **MOMENT:** Encountering an empty list: the Workflows review queue with nothing pending, the Active Chat Queue with no customers, the ‘Pinned Resources’ panel with nothing pinned.
* **SCREEN-ELEMENT:** Workflows: centered checkmark icon + ‘All clear! No articles are waiting for review or approval.’ Active Chat Queue: plain text ‘No cases in this queue.’ with no icon. Pinned Resources: a one-line muted hint, also with no icon.
* **WHAT HAPPENS:** Three structurally identical ‘there is nothing here’ situations are presented in three different styles — one polished, two minimal — with no shared icon, weight, or tone.
* **WHAT SHOULD HAPPEN:** Adopt the Workflows treatment (icon + bold headline + muted explanatory line) as the one standard empty-state pattern and reuse it everywhere.
* **IMPACT:** Inconsistent empty states make the product feel stitched together from several screens rather than one coherent system — a subtle but real polish signal during a demo.
* **SEVERITY:** LOW

### C2 — Error and success feedback are inconsistent in both presence and presentation

* **PERSONA:** Cross-cutting (Admin, Agent)
* **MOMENT:** Comparing how the product communicates failure vs. success: a blank-Title save, a missing-team submit error, a successful gap report, a successful ‘Copy to Chat.’
* **SCREEN-ELEMENT:** Save Changes (silent failure), team-assignment error banner (loud, persistent), ‘Gap reported successfully’ (small inline confirmation), ‘Copy to Chat’ (toast).
* **WHAT HAPPENS:** Successes consistently get clear, appropriately-sized feedback. Failures are inconsistent: some get a loud, persistent red banner, others get nothing at all. There is no single, predictable place to look to know something went wrong.
* **WHAT SHOULD HAPPEN:** Standardize on one feedback pattern (toast + inline field highlight) for every form error, applied uniformly, not selectively.
* **IMPACT:** Whether an action fails loudly, fails silently, or succeeds quietly is currently unpredictable from the user's point of view — directly damaging trust in the moments (saving, approving, publishing) where it matters most.
* **SEVERITY:** HIGH

### C3 — Sidebar does not collapse at mobile width (partially verified)

* **PERSONA:** Cross-cutting (mobile / responsive)
* **MOMENT:** Resizing the browser to a 375px-wide viewport to simulate an agent checking the KB on a phone mid-call.
* **SCREEN-ELEMENT:** Login screen vs. Agent workspace persistent left sidebar (Dashboard / Customer Chat / Knowledge Base / Gap Reports / Glossary).
* **WHAT HAPPENS:** The public login screen reflowed correctly into a centered, single-column layout at 375px. The Agent workspace sidebar appeared to remain a fixed-width column with no observed collapse behavior, but a clean, fully confirmed screenshot of the in-app workspace at 375px could not be captured before the testing session needed to reconnect — this should be re-verified directly before relying on it.
* **WHAT SHOULD HAPPEN:** Collapse the sidebar to an icon rail or off-canvas drawer below a defined breakpoint (e.g. 768px) so search and article content get full width on a phone.
* **IMPACT:** Field agents on a tablet or phone are a plausible real user; a non-collapsing sidebar eats into already-limited screen space exactly when it's most scarce. Flagged for re-verification rather than treated as fully confirmed.
* **SEVERITY:** MEDIUM

### C4 — Arabic content is not discoverable and result layout does not mirror for RTL

* **PERSONA:** Cross-cutting (RTL / Arabic)
* **MOMENT:** Searching with Arabic text and trying to locate the existing Arabic-titled article referenced in the platform's own analytics.
* **SCREEN-ELEMENT:** Knowledge Base search input and result cards.
* **WHAT HAPPENS:** Arabic text entered into the search box displays correctly right-to-left inside the input itself, but the surrounding UI — search icon position, result cards, category chips — stays fully left-to-right with no mirroring, and the search fails to surface the matching Arabic article even on an exact-title query (see Persona 1 finding).
* **WHAT SHOULD HAPPEN:** Support RTL layout for result cards and article bodies when Arabic content is present, and make Arabic content reliably discoverable through Arabic search, browse, or a language filter.
* **IMPACT:** For a market where Arabic is the primary language, content that exists but cannot be found defeats the purpose of localizing it — this should be treated as a launch blocker for Iraq, not a cosmetic RTL detail.
* **SEVERITY:** HIGH

### C5 — Red/amber badge styling is reused for both neutral and negative statuses

* **PERSONA:** Cross-cutting (status badge color coding)
* **MOMENT:** Scanning the Gaps Queue and Articles Manager in quick succession, the normal admin triage workflow.
* **SCREEN-ELEMENT:** ‘NEW’ status badge (Gaps Queue) and ‘REJECTED’ status badge (Articles Manager), both rendered in a similar red/amber tone.
* **WHAT HAPPENS:** A freshly-reported, not-yet-triaged gap (‘NEW’) and content that was actively reviewed and rejected (‘REJECTED’) share near-identical badge coloring, despite one being neutral and the other a negative outcome.
* **WHAT SHOULD HAPPEN:** Reserve red for negative/blocked outcomes only (Rejected, Error); use a distinct neutral color (blue or gray) for ‘New/unclaimed’ items.
* **IMPACT:** An admin triaging a long queue at speed could misread urgency or nature of an item from color alone — the entire point of a color-coded status system.
* **SEVERITY:** LOW

# PRIORITY DESIGN CHANGES FOR THE DEVELOPER

Sorted by severity. Each item references the finding(s) it corresponds to above.

### CRITICAL

1. Fix Super Admin sidebar navigation — every item (Organizations, User Accounts, Teams Manager, Articles Manager, Gaps Queue, Workflows, Audit Logs, Glossary) is dead; Salman cannot leave the Dashboard. [F3.1]
2. Fix or remove the search match-percentage scoring — it returns ≈10–24% for every query regardless of relevance, at both the Agent and platform level. [F1.2, F3.2]
3. Surface the self-approval governance block at the point of the failed click in the Articles Manager list view, not only one click deeper on the review screen. [F2.2]
4. Scope ‘Top Search Queries’ on the Admin Analytics Dashboard to the logged-in tenant — it currently shows another org's data inside OODI's own view. [F2.5]

### HIGH

1. Make the article-detail back-link return to the actual prior search results (query + scroll position), not the Dashboard. [F1.1]
2. Fix Arabic-language search matching and make existing Arabic content discoverable through search or browse. [F1.3, C4]
3. Add inline validation (red field highlight + message) when Save is clicked with a missing required field such as Title. [F2.1]
4. Standardize one feedback pattern (toast + inline highlight) for every form error across the app, applied consistently rather than selectively. [C2]
5. Add diagnostic context (tooltip or drill-down) to the platform-wide confidence metric so a low number is explainable, not just alarming. [F3.2]

### MEDIUM

1. Fix trend-arrow direction/color logic on all KPI tiles so ‘down’ consistently means worse and ‘up’ consistently means better, regardless of the metric. [F1.4]
2. Clear stale error banners automatically once the underlying issue is fixed and the form is resaved. [F2.3]
3. Reconcile ‘Pending’ vs. ‘In Review’ terminology across Articles Manager, Workflows, and the status dropdown into one term. [F2.4]
4. Fix Audit Log actor attribution so customer-triggered events are logged under the customer, not the admin viewing the page. [F2.6]
5. Stop internal dashboard page-loads from incrementing the public ‘Total Views’ counter. [F3.3]
6. Re-verify and fix sidebar collapse behavior on the Agent workspace below ~768px width. [C3]

### LOW

1. Standardize empty-state treatment (icon + headline + muted subtext) across Workflows, Active Chat Queue, and Pinned Resources. [C1]
2. Reserve red/amber badge styling for negative outcomes only; give ‘New/unclaimed’ items a distinct neutral color from ‘Rejected.’ [C5]

# THE ONE THING

**Fix the Super Admin sidebar navigation before anything else. [F3.1]**

The Super Admin role is the persona a client will most associate with ‘is this enterprise-ready?’ — and right now it cannot navigate anywhere beyond the Analytics Dashboard. Every sidebar link, across both the platform-management section (Organizations, User Accounts, Teams Manager) and the content-oversight section (Articles Manager, Gaps Queue, Workflows, Audit Logs, Glossary), fails to route. This was confirmed three independent ways — mouse clicks, accessibility-tree reference clicks, and a full DOM read of the navigation component — ruling out a tool artifact and confirming a genuine, reproducible product bug.

Organizations is the natural first click for a Super Admin trying to demonstrate platform oversight to a client. If that click is made live during the demo — and it almost certainly will be — the demo stalls immediately on what looks like a broken application, before any of the genuinely good parts of the product (the workflow engine, the separation-of-duties governance logic, the gap-reporting loop) are ever shown.

Every other finding in this report is worth fixing on its own merits, but none of them will matter if this is the first thing the room sees.

# MEETING NOTES: ZAIN - DAILY SYNC 2.0 - JUNE 25

**Meeting Purpose:** Review the Super Admin, Admin, and Agent dashboards for user flow and UX improvements.[cite: 2]

## Key Takeaways
* **Analytics must be actionable:** Dashboards must enable direct action (e.g., clicking a low-rated article to edit it), not just display data.[cite: 2]
* **Data must be segmented:** Analytics must distinguish between agent and public (customer) activity to provide meaningful insights and identify knowledge gaps.[cite: 2]
* **UX requires immediate fixes:** The UI has critical issues, including unreadable icons, misleading arrow indicators, and inconsistent labeling.[cite: 2]
* **Prioritize user flow over implementation:** The team must first define the ideal user flow for each persona, then determine the necessary backend changes.[cite: 2]

## Topics

### Core Problem: Analytics Are Not Actionable
* The primary issue is that analytics are not actionable[cite: 2]. Users cannot drill down from high-level metrics to the specific articles or actions needed to resolve issues[cite: 2].
* *Example:* A Super Admin sees a "Helpful Rate" metric but cannot click it to view and fix the "unhelpful" articles[cite: 2].
* *Solution:* Dashboards must be designed to guide users from insight to action[cite: 2].

### Required Data Segmentation: Agent vs. Public
* To provide meaningful insights, analytics must distinguish between agent and public (customer) activity[cite: 2].
* *Rationale:* The system needs to identify knowledge gaps for both audiences[cite: 2].
* *Implementation:* A toggle (e.g., "Analytics: Total | Customer | Agent") is needed to filter all dashboard data[cite: 2].
* *Note:* This is a core requirement, not a future-phase feature[cite: 2].

### UX & UI Improvements
* **Visuals:**
    * **Icons:** Super Admin icons are unreadable (gray on white); font size and color contrast need adjustment[cite: 2].
    * **Arrows:** A red arrow pointing down for "45 new this week" is misleading; "new" is a positive event and should use an upward arrow[cite: 2].
    * **Headers:** Empty space in dashboard headers (e.g., "Article Performance") should be removed[cite: 2].
* **Labeling:**
    * **"Resolution Rate":** Should be renamed to "Resolved" to reflect a status, not a rate[cite: 2].
    * **"Knowledge Gaps Resolved":** The metric's source (e.g., agent submissions) must be clarified[cite: 2].
* **Data Integrity:** Mock data is inconsistent (e.g., "316 this week" appears in multiple unrelated widgets), which undermines trust in the system[cite: 2].

### Article Visibility & Workflow
* **Visibility Options:** The system requires clear visibility settings for articles:[cite: 2]
    * Public (Customers)[cite: 2]
    * Agents (can be team-specific)[cite: 2]
    * Admins (can be team-specific)[cite: 2]
* **Workflow Governance:** The "Submit for Review" option must be available for all users, including Admins and Super Admins, to enforce workflow governance[cite: 2].
